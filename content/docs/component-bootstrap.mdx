A "component" in Gibson is the umbrella for **agents**, **tools**, and
**plugins**. They register with Gibson by name and are addressed by name
at dispatch time. They share one identity model on the wire — every call
to Gibson carries a Bearer JWT — but they take two different routes to
mint that JWT, because the audiences and lifecycles are different.

## The two enrollment paths

| Kind | Bootstrap | Long-lived credential | Per-call token |
|---|---|---|---|
| **Plugin** | One-shot bootstrap token, single-use, 24 h expiry | Persistent **Ed25519 host key** at `~/.gibson/plugin/<name>/host_key` (mode 0600) | Plugin signs a fresh per-RPC token (55 s expiry) |
| **Agent / Tool** | Dashboard-issued **OAuth2 `client_id` + `client_secret`** (one-time) | Credentials JSON at `~/.gibson/<kind>/credentials` (mode 0600) | OAuth2 `client_credentials` grant returns a JWT (~1 h expiry, refreshed 5 min before TTL) |

Why two paths? Plugins are admin-installed service integrations with
manifest-declared secret bindings — the bootstrap-token-then-host-key
pattern matches that workflow, and per-RPC ephemeral signing minimises
the replay window. Agents and tools are user-built executables that
need **standard OIDC compatibility** so third-party SDKs and frameworks
can talk to Gibson — OAuth2 `client_credentials` is the right protocol
for that audience.

The dashboard's **deploy wizard** at `/dashboard/deploy` handles both
paths via a per-type dispatcher. Pick the kind in step 1, the wizard
routes you to the right flow:

- `kind=plugin` walks through manifest upload, secret bindings, and
  atomic register, ending with a bootstrap token and a
  `gibson component register --kind plugin --token <T>` command to run
  on the host you'll deploy from.
- `kind=agent` or `kind=tool` walks through type/name selection, the
  **Permissions** step (per-action grants on components, plus plugin
  invocation grants for tools), credential issuance, and a polling
  step that confirms the component connects.

## Per-call protocol

Each component-to-Gibson RPC is a standard authenticated gRPC call:

- Plugins sign a fresh short-lived token per RPC with their host key
  and send it as a Bearer credential. Gibson validates the signature
  against the host key recorded at enrolment.
- Agents and tools use **OAuth2 `client_credentials`** to exchange
  their `client_id` / `client_secret` for a JWT (about an hour of
  TTL), then present that JWT as a Bearer credential on every call.
  The SDK refreshes the token automatically, ~5 minutes before
  expiry.

The SDK handles all of this transparently — your code calls
`agent.New(...)` (or the equivalent for tools and plugins), passes a
context, and Gibson sees a fully-authenticated request. You do not
write any of the token plumbing yourself.

## Glossary

| Term | What it is |
|---|---|
| **Bootstrap token** | Single-use credential the plugin path mints during dashboard register. 24 h expiry. Consumed once by `gibson component register --kind plugin`, then discarded. |
| **Host key** | Persistent Ed25519 keypair stored on disk at `~/.gibson/plugin/<name>/host_key` (mode 0600). Identifies a plugin install. Public-key thumbprint is the host_id. |
| **Agent key** | Ephemeral Ed25519 keypair regenerated per-process. Signs the per-RPC token; chained to the host key in the token's `cnf` claim. |
| **client_id / client_secret** | OAuth2 credentials the dashboard issues for agents/tools through the deploy wizard. The secret is shown exactly once. |
| **Capability-grant token (CG-JWT)** | Per-mission task token Gibson mints, scoped to a specific RPC set. Presented alongside the primary credential during mission execution. |
| **Component grant** | Per-action read / configure / execute permission on a registered component, granted via the dashboard's Permissions tab. |
| **Plugin invocation grant** | Binary `can_invoke` permission on a plugin. Only **tools** can hold this; agents do not directly invoke plugins. |

## Troubleshooting a 401

A 401 means Gibson rejected your component's credential. The customer
debug surface for this is the **dashboard** plus `gibson inspect` — you
should not need to read any server-side logs.

1. **Verify the credential file is intact.** Mode must be 0600. The
   SDK refuses to load a credential file with looser permissions and
   surfaces a clear error if you try.
2. **Run `gibson inspect`.** Auto-detects the local credentials,
   exchanges them for a token, calls Gibson's identity service, and
   prints who you are along with the permissions you hold.
3. **If `inspect` fails at the token exchange step**, your credential
   was rejected at issuance time — re-issue from the dashboard's
   deploy wizard. (`client_secret` and bootstrap tokens are one-time;
   if you lost them or they expired, issuing a fresh credential is
   the path forward.)
4. **If `inspect` succeeds but a specific RPC still gets denied**,
   you have a credential but not the grant the RPC needs. Open the
   relevant component's detail page in the dashboard and check the
   **Permissions** tab — the missing grant will be listed there with
   a one-click toggle to add it (subject to your own role's
   permissions to edit grants).

If you've worked through these and the call still fails, the credential
issuance flow has misfired — file a bug with the output of
`gibson inspect --json` so we can correlate it on our side.
