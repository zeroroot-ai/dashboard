[Taxonomy](/docs/taxonomy) answers *"is this a valid node?"* Ontology
answers *"is `CWE-79` a kind of injection? does `SOC2 CC6.1` roll
up to `CC6`?"* It's pure read-side enrichment on top of the typed
graph, the write path is unchanged.

## When you need it

- Mapping your custom finding categories to industry vocab
  (`mycorp:HardcodedAWSKey ⊑ cwe:CWE-798`) so queries on the
  industry term include your nodes.
- Declaring sub-control relationships in a compliance framework so
  filtering by `SOC2 CC6` returns everything under `CC6.1`, `CC6.2`, …
- Telling Gibson that two scanners' outputs refer to the same host
  or artifact (identity-forming properties, see IFPs below).
- Bridging your taxonomy to MITRE ATT&CK tactics so the dashboard's
  attack-pattern views pick up techniques your component contributes.

If you have none of these needs, you don't need an `ontology.yaml`.
Taxonomy alone is enough.

## The four building blocks

- **Prefixes**, short labels for the IRIs of external vocabularies
  (`cwe:`, `attack:`, `asvs:`, `mycorp:`). One line each.
- **Hierarchies**, `subClassOf` edges. *"`mycorp:LeakedSecret` is a
  kind of `mycorp:ProprietaryFinding`."*
- **Equivalences**, `sameAs` / `equivalentClass` edges. *"`mycorp:LeakedSecret`
  is the same class as `gibson:disclosure`."* Queries on either side
  pick up nodes labelled with the other.
- **IFPs (identity-forming properties)**, the property set that
  uniquely identifies a custom node. Lets Gibson deduplicate when
  two components emit the same conceptual entity.

## YAML shape

A minimal `ontology.yaml`:

```yaml
version: "0.1.0"
prefixes:
  mycorp: "https://mycorp.example/sec/"
hierarchies:
  - parent: mycorp:ProprietaryFinding
    children: [mycorp:LeakedSecret, mycorp:HardcodedKey]
equivalences:
  - [mycorp:LeakedSecret, gibson:disclosure]
ifps:
  - node_type: my_custom_artifact
    properties: [content_hash]
```

Drop this at the root of your component repo, next to `taxonomy.yaml`.

## Build flow

1. You write `ontology.yaml`.
2. `gibson component build` validates it (well-formed YAML, every
   IRI prefix is declared, no cycles in `hierarchies`, every IFP
   `node_type` exists in `core.yaml` or your `taxonomy.yaml`, no
   `equivalences` conflict with an extension already registered for
   the tenant).
3. The ADK emits `gen/ontology_extension.go`, an `OntologyExtension`
   value baked into your component binary.
4. The component registers the extension with the daemon at component registration.
5. The daemon's reasoner merges it into the tenant's closure and
   rebuilds derived indices.

## What lights up in the dashboard

- A dashboard user filters findings by **`SOC2 CC6`**. Your custom
  finding category mapped under `CC6.2` now appears in the list -
  no UI change required on your side.
- A user opens the **MITRE ATT&CK** matrix. Custom techniques you
  declared as `subClassOf` an existing tactic light up the
  corresponding cell.
- The **deduplication** view collapses two findings from different
  scanners that share an IFP-defined identity, instead of showing
  them as separate rows.
- Knowledge-graph queries that walk `subClassOf` (`h.QueryNodes`
  with hierarchy expansion) return your custom subclasses
  transparently.

## Failure modes

The validator catches these before your component reaches the daemon.
What to do with each:

| Error | What it means | Fix |
|---|---|---|
| `cycle in hierarchies` | Your `subClassOf` chain loops back on itself. | Read the cycle the validator prints. Drop one edge or restructure so the chain is a DAG. |
| `unknown prefix: <p>` | You used `<p>:Foo` without declaring `<p>` in `prefixes`. | Add the prefix-to-IRI mapping. Use the canonical IRI for industry vocab (`cwe:` → `https://cwe.mitre.org/data/definitions/`, etc.). |
| `IFP property <prop> not on node_type <n>` | The IFP references a property that doesn't exist on the named node type. | Either add the property in `taxonomy.yaml` first, or fix the IFP entry. Run `gibson component build` once after the taxonomy edit to refresh validation. |
| `conflicting equivalence: <A> <-> <B>` | Another extension already registered an equivalence touching `<A>` or `<B>` that contradicts yours. | Decide whether your component should override or follow the existing mapping. If override, coordinate with the tenant admin, the daemon won't silently merge contradictions. |

When an assistant hits one of these, the right move is to read the
validator output verbatim, fix the YAML, and rerun the build. Don't
`--force` past it; the daemon will reject a non-validating extension
at registration anyway.

## Power-user note

A raw `.ttl` (Turtle) file can sit next to `ontology.yaml` for
axioms the YAML shape doesn't express (e.g. property restrictions,
cardinality, disjointness). The ADK passes it through to the daemon's
reasoner unchanged. You don't need this for the common cases above -
it exists for teams already invested in OWL.

## Prompting your assistant

- *"Add the OWASP ASVS L1 control hierarchy under prefix `asvs:`,
  pointing at `https://owasp.org/www-project-application-security-verification-standard/asvs/`.
  Then map our existing `gibson:input-validation` finding category as
  `equivalentClass` to `asvs:V5`. Run `gibson component build` and
  paste any validator errors."*
- *"Open `ontology.yaml` and declare that our custom
  `mycorp:HardcodedAWSKey` is a `subClassOf` `cwe:CWE-798`. Use the
  existing `cwe:` prefix if it's already there; otherwise add it."*
- *"In `ontology.yaml`, add an IFP for our `my_custom_artifact` node
  type using `content_hash` as the identity property. Confirm
  `content_hash` is declared in `taxonomy.yaml` before you save."*
