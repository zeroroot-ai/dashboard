This page gets your workstation talking to Gibson. Gibson itself is a
hosted service, there's nothing to install on the server side. The only
thing you install locally is the **`gibson`** binary (shipped by the
ADK), which you'll use to scaffold and register your agents, tools, and
plugins.

## Prerequisites

| Tool | Why |
|---|---|
| **Go** (the version listed in the SDK's `go.mod`) | Building agents, tools, and plugins |
| **`gibson`** | Scaffolding, registration, plugin runtime |
| **Docker** | Optional, only needed if you'll run plugins as pods |
| **`protoc`** | Optional, only if you're defining custom tool/plugin protos |

## 1. Sign up for a Gibson tenant

Go to [zeroroot.ai](https://zeroroot.ai) and create an account. Your
tenant is provisioned automatically; you'll be the **owner** of it.

Trial accounts include 14 days of full platform access, see
[zeroroot.ai/pricing](https://zeroroot.ai/pricing).

## 2. Install the `gibson` CLI

The fastest path:

```bash
go install github.com/zeroroot-ai/adk/gibson/cmd/gibson@latest
```

Or build from source if you want to track `main`:

```bash
git clone https://github.com/zeroroot-ai/adk.git
cd adk/gibson
go build -o ./bin/gibson ./cmd/gibson
export PATH="$PWD/bin:$PATH"
```

Verify:

```bash
gibson --help
```

You should see the top-level command groups: `init`, `login`, `logout`,
`component`, `connector`, `agent`, `mission`, `target`, `provider`,
`inspect`, and `docs`.

## 3. Point at your tenant

Run `gibson init` once per workspace. It writes `.gibson/workspace.yaml`,
pinning the platform URL so you don't pass it on every command:

```bash
gibson init --gibson-url https://api.zeroroot.ai
```

Use `--global` to write `~/.gibson/workspace.yaml` instead, so every
directory shares one default. The CLI and SDK also accept `GIBSON_URL`
from the environment, which overrides the workspace file when set.

## 4. Sign in

Authenticate the CLI as yourself with `gibson login`. It runs the OAuth
2.0 Device Authorization Grant: the CLI prints a URL and a short code,
you approve in a browser, and the session is stored at
`~/.gibson/auth/credentials` (mode `0600`). Every later `gibson` command
then acts as you, and the session refreshes silently.

```bash
gibson login
```

The CLI discovers the identity issuer from the platform automatically.
Run `gibson logout` to end the session.

## 5. Enroll a component

Enrollment mints a **single-use bootstrap token** for a machine
identity. Every kind (agent, tool, plugin) enrolls the same way, and the
token is the only credential the component needs to bootstrap, there is
no client secret anywhere.

```bash
gibson agent enroll --name my-first-agent --kind agent
```

The token is printed once and expires in 24 hours. Copy it, then
exchange it for a persistent runtime credential. Run `register` from the
component directory (scaffolded by `gibson component init`); it reads the
kind and name from `component.yaml`:

```bash
gibson component register --token <bootstrap-token>
```

You can also pass the token on stdin (`--token -`) or via
`GIBSON_BOOTSTRAP_TOKEN`, and override the auto-detected kind, name, or
directory with `--kind`, `--name`, and `--dir`.

`register` runs the capability-grant handshake and writes two files at
mode `0600`: an Ed25519 **host key** at
`~/.gibson/<kind>/<name>.host_key` and a **runtime credential** at
`~/.gibson/<kind>/<name>.runtime.json`. Treat both like private keys,
they identify your component to Gibson.

If registration fails, see
[Component bootstrap & auth](/docs/component-bootstrap#troubleshooting-a-401)
for the layered checks.

## 6. Verify with `inspect`

```bash
gibson inspect              # auto-detects the local credential
gibson inspect --kind agent --name my-first-agent
gibson inspect --json       # raw output, scriptable
```

`inspect` reads your local runtime credential, calls Gibson's identity
service, and prints who you are along with the permissions you hold. If
`inspect` succeeds, your credential is healthy and you can move on.

## Storing credentials safely

Credential files live in `~/.gibson/` and are restricted to mode `0600`
by `register`. Don't commit them. For CI and production deployments:

- **Headless runtimes**: instead of writing files, pass the runtime
  credential as `GIBSON_AGENT_KEY` (the base64-encoded credential) and
  the daemon endpoint as `GIBSON_URL`, sourced from a secret. The SDK
  picks them up automatically. This is the same for every component
  kind.
- **Fresh runtime instances**: bake the host key into the deployment
  artifact, or re-register on each instance using a freshly-issued
  bootstrap token.
- **Workload identity**: if your runtime exposes a workload-identity
  socket, the SDK auto-detects it and uses mTLS alongside the bearer
  credential. No extra configuration needed.

## Multi-environment setups

If you operate against more than one Gibson tenant (e.g. staging +
production), point `GIBSON_URL` at the right tenant per shell session
or per CI job, and keep credentials separated by setting `GIBSON_HOME`
to a distinct config dir per tenant (or by supplying a per-tenant
`GIBSON_AGENT_KEY` in headless runtimes).
