`gibson` is the local binary you use to scaffold components, register
them with your tenant, validate manifests, build and run components,
submit missions, and verify credentials.

Modern CLI is unified under five top-level groups:

| Group | Purpose |
|---|---|
| `init` | Initialise a Gibson workspace. |
| `component` | Scaffold, build, register, validate, and run agents / tools / plugins. |
| `mission` | Author, validate, render, and submit missions. |
| `inspect` | Print who you are and what permissions your local credential carries. |
| `docs` | Emit machine-readable docs (JSON Schemas, etc.). |

> **Legacy verbs.** The per-kind verbs (`gibson agent enroll`,
> `gibson tool enroll`, `gibson plugin enroll`, `gibson plugin run`,
> …) still work for backwards-compatibility — see the
> [Legacy shims](#legacy-shims) appendix at the bottom of this page.
> New work should use the unified `gibson component <verb>` form.

Global flags available on every command:

| Flag | Default | What it does |
|---|---|---|
| `--gibson-url` | `$GIBSON_URL` | Override the Gibson endpoint for one invocation. |
| `--credentials-path` | `~/.gibson/<kind>/credentials` | Use credentials from a non-default location. |
| `--quiet` | off | Suppress progress output; only print errors. |
| `--json` | off | Emit machine-readable output where supported. |

## `gibson init`

Initialise a Gibson workspace in the current directory. Sets up the
shared scaffolding (`.gibson/`, default manifests) that component-init
commands expect.

```bash
gibson init
```

## `gibson component`

Group for the component lifecycle. Replaces the legacy per-kind verbs.

### `gibson component init`

Scaffold a new component directory. The `--kind` flag picks the
component shape (agent / tool / plugin); the generated files differ
per kind.

```bash
gibson component init my-agent     --kind agent
gibson component init my-portscan  --kind tool
gibson component init my-shodan    --kind plugin --dir ./my-shodan
```

| Flag | Default | Notes |
|---|---|---|
| `--kind` | yes | One of `agent`, `tool`, `plugin`. |
| `--dir` | `.` | Target directory. Created if missing. |
| `--with-secret` | none | (Plugin) Pre-populate `spec.secrets` with one entry of the given name. |
| `--force` | off | Overwrite existing files in the target directory. |

For plugins the scaffold includes `plugin.yaml`, `main.go`, a sample
proto, `Makefile`, `Dockerfile`, `.gitignore`, and `README.md`. Names
must match `^[a-z][a-z0-9-]{0,61}[a-z0-9]$` (DNS-label style).

### `gibson component build`

Generate, validate, and compile the component binary in the current
directory. Reads the component manifest, runs proto codegen, then
`go build`.

```bash
gibson component build
```

This is the canonical "build my component" command. CI pipelines that
used to invoke `make build` should call `gibson component build`
instead — the ADK keeps the toolchain pinned, so the build output is
reproducible across machines.

### `gibson component register`

Register a component install with your tenant. For agents and tools,
exchanges OAuth2 client credentials issued by the dashboard's deploy
wizard for a credentials file. For plugins, exchanges a single-use
bootstrap token for a persistent Ed25519 host key.

```bash
# Agents and tools
gibson component register --kind agent \
  --client-id     <id> \
  --client-secret <secret> \
  --name          <component-name>

gibson component register --kind tool \
  --client-id     <id> \
  --client-secret <secret> \
  --name          <component-name>

# Plugins
gibson component register --kind plugin --token <bootstrap-token>
```

| Flag | Required | Notes |
|---|---|---|
| `--kind` | yes | `agent`, `tool`, or `plugin`. |
| `--client-id` | agent / tool | OAuth2 client ID from the deploy wizard. |
| `--client-secret` | agent / tool | OAuth2 client secret. Shown exactly once at issuance. |
| `--name` | agent / tool | Component name (kebab-case, unique within tenant). |
| `--token` | plugin | Single-use bootstrap token from the deploy wizard. |
| `--credentials-path` | no | Override the credentials file location. |

What it does:

1. Writes a credentials file at `~/.gibson/<kind>/credentials` (mode
   `0600`). For plugins, also writes an Ed25519 host key at
   `~/.gibson/plugin/<name>/host_key` (mode `0600`).
2. Mints a JWT (or signs a capability-grant for plugins) to verify
   the credential.
3. Calls Gibson's identity service to confirm registration.
4. Notifies the dashboard's deploy wizard so it can advance.

If verification fails, the command exits non-zero and the file is
removed. See
[Component bootstrap & auth → Troubleshooting a 401](/docs/component-bootstrap#troubleshooting-a-401).

### `gibson component validate`

Validate the component manifest in the current directory against the
schema, without contacting Gibson.

```bash
gibson component validate
```

Useful in CI to catch manifest errors before they reach the
register step. For plugins specifically, `gibson plugin validate`
remains a legacy shim if you need it.

### `gibson component run`

Run the built component binary. Equivalent to invoking the binary
directly (which calls `agent.Serve`, `tool.Serve`, or `plugin.Serve`).

```bash
gibson component run
```

| Flag | Default | Notes |
|---|---|---|
| `--manifest` | auto-detect | Use a non-default manifest path. |

Useful in development; production deployments typically invoke the
binary directly under their process manager (systemd, Kubernetes, …).

## `gibson mission`

Group for mission authoring, validation, rendering, and submission.

### `gibson mission submit`

Submit a mission file to the daemon. The file format is auto-detected
from the extension — `.cue`, `.yaml`, `.yml`, and `.json` are all
supported.

```bash
gibson mission submit missions/recon.yaml --target example.com
gibson mission submit missions/triage.cue --target https://app.example.com
```

| Flag | Required | Notes |
|---|---|---|
| `--target` | (depends on mission) | Target reference (URL, host, k8s cluster, …). The mission may declare its own default. |
| `--variables` | no | Path to a variables file or inline `key=value` pairs. |

What it does:

1. Loads and parses the mission file (CUE / YAML / YML / JSON).
2. Validates against the `MissionDefinition` schema.
3. Submits to the daemon's `CreateMission` RPC.
4. Calls `RunMission` and streams the lifecycle events to stdout.

See [Missions](/docs/missions) for the full lifecycle.

## `gibson inspect`

Print your effective identity and permissions.

```bash
gibson inspect                            # auto-detect
gibson inspect --kind agent               # specify kind
gibson inspect --kind plugin --name foo   # specify name when several plugins are registered locally
gibson inspect --json                     # machine-readable
```

| Flag | Default | Notes |
|---|---|---|
| `--kind` | auto-detect | `agent`, `tool`, or `plugin`. Required when more than one kind has a local credential. |
| `--name` | auto | Required when more than one credential exists for a single kind. |
| `--json` | off | Emit the raw identity payload. |

What it does:

1. Loads the credential.
2. Mints a JWT (or signs a capability-grant for plugins).
3. Calls Gibson's identity service to fetch your principal.
4. Prints your identity, tenant, and effective grants.

If `inspect` fails:

- **At credential load** — credential file is missing or has wrong
  permissions. Fix the file mode (`chmod 600`) or re-register.
- **At token mint** — credential is malformed. Re-register.
- **At identity call** — Gibson rejected the credential (revoked,
  expired, wrong tenant). Re-issue from the dashboard.

## `gibson docs`

Emit machine-readable docs — JSON Schemas for the component manifest,
mission file format, taxonomy types, and similar. Useful for IDE
plugins, codegen, and pinning your CI against a specific CLI version's
contracts.

```bash
gibson docs
```

## Exit codes

| Code | Meaning |
|---|---|
| `0` | Success. |
| `1` | Generic error. |
| `2` | Bad arguments / usage. |
| `3` | Credential file missing or unreadable. |
| `4` | Credential rejected by Gibson. |
| `5` | Network error reaching Gibson. |
| `75` | Plugin runtime exited cleanly to be restarted (rotation-restart sentinel). |

CI scripts can rely on these. `2` and `3` indicate operator error;
`4` and `5` indicate platform-side issues to surface.

## Environment variables

| Variable | Used by | Effect |
|---|---|---|
| `GIBSON_URL` | every command, every component | Endpoint to dial. |
| `OIDC_CLIENT_CREDENTIALS_CLIENT_ID` | agents, tools | Inline client ID — no credentials file needed. |
| `OIDC_CLIENT_CREDENTIALS_CLIENT_SECRET` | agents, tools | Inline client secret. |
| `OIDC_CLIENT_CREDENTIALS_TOKEN_URL` | agents, tools | Token endpoint when env-based credentials are in use. |
| `GIBSON_CREDENTIALS_PATH` | every command | Override the default credentials path. |
| `GIBSON_LOG_LEVEL` | every command | `debug` / `info` / `warn` / `error`. Default `info`. |

The env-variable path is the recommended way to ship CI workloads —
no credential files on disk.

## Legacy shims

The per-kind verbs from earlier CLI versions still work for
backwards-compatibility. New work should use `gibson component <verb> --kind <kind>`,
but existing scripts and CI jobs don't have to change.

| Legacy form | Modern equivalent |
|---|---|
| `gibson agent enroll …` | `gibson component register --kind agent …` |
| `gibson tool enroll …` | `gibson component register --kind tool …` |
| `gibson plugin enroll --token …` | `gibson component register --kind plugin --token …` |
| `gibson plugin init …` | `gibson component init --kind plugin …` |
| `gibson plugin validate` | `gibson component validate` (in a plugin directory) |
| `gibson plugin run` | `gibson component run` |

The shims dispatch to the same underlying logic; flag semantics are
identical.
