This page gets your workstation talking to Gibson. Gibson itself is a
hosted service — there's nothing to install on the server side. The only
thing you install locally is the **`gibson`** binary (shipped by the
ADK), which you'll use to scaffold and register your agents, tools, and
plugins.

## Prerequisites

| Tool | Why |
|---|---|
| **Go** (the version listed in the SDK's `go.mod`) | Building agents, tools, and plugins |
| **`gibson`** | Scaffolding, registration, plugin runtime |
| **Docker** | Optional — only needed if you'll run plugins as pods |
| **`protoc`** | Optional — only if you're defining custom tool/plugin protos |

## 1. Sign up for a Gibson tenant

Go to [zero-day.ai](https://zero-day.ai) and create an account. Your
tenant is provisioned automatically; you'll be the **owner** of it.

Trial accounts include 14 days of full platform access — see
[zero-day.ai/pricing](https://zero-day.ai/pricing).

## 2. Install the `gibson` CLI

The fastest path:

```bash
go install github.com/zero-day-ai/adk/cmd/gibson@latest
```

Or build from source if you want to track `main`:

```bash
git clone https://github.com/zero-day-ai/adk.git
cd adk/cli
go build -o ./bin/gibson ./cmd/gibson
export PATH="$PWD/bin:$PATH"
```

Verify:

```bash
gibson --help
```

You should see the top-level command groups: `init`, `component`,
`docs`, `inspect`, `mission`.

## 3. Point at your tenant

Set `GIBSON_URL` in your shell:

```bash
export GIBSON_URL=https://api.zero-day.ai
```

Add it to your shell profile (`~/.zshrc`, `~/.bashrc`, …) so it
persists across sessions. The SDK reads `GIBSON_URL` from the
environment when your component dials Gibson; the value can also be
embedded in the credentials file written by `gibson component register`.

## 4. Get a credential from the dashboard

Open the dashboard at
[`app.zero-day.ai/dashboard/deploy`](https://app.zero-day.ai/dashboard/deploy).
This is the **deploy wizard** — the one place where component identities
are issued.

1. Pick a kind: **Agent**, **Tool**, or **Plugin**.
2. Name your component (kebab-case, unique within your tenant).
3. **Permissions** — grant the component access to the data and
   capabilities it needs. See [Roles & permissions](/docs/rbac).
4. **Issue credentials**:
   - Agents and tools get an OAuth2 `client_id` + `client_secret`. The
     **secret is shown exactly once** — copy it before you leave the
     page.
   - Plugins get a single-use **bootstrap token** (default 1 hour,
     configurable up to 24 hours).
5. **Connect** — leave the wizard open. It polls until your local
   `gibson` reports back that the component connected.

## 5. Register on your machine

The modern CLI uses a single unified `component register` verb,
parameterised by `--kind`:

```bash
# Agents
gibson component register --kind agent \
  --client-id     <from-wizard> \
  --client-secret <from-wizard> \
  --name          my-first-agent

# Tools
gibson component register --kind tool \
  --client-id     <from-wizard> \
  --client-secret <from-wizard> \
  --name          my-first-tool

# Plugins
gibson component register --kind plugin --token <from-wizard>
```

Each command writes a credential file at `~/.gibson/<kind>/credentials`
(file mode `0600`). For plugins, an Ed25519 host key is also written
to `~/.gibson/plugin/<name>/host_key` (mode `0600`). Treat both like
private keys — they identify your component to Gibson.

> **Legacy shims.** The older per-kind verbs (`gibson agent enroll`,
> `gibson tool enroll`, `gibson plugin enroll`) still work for
> backwards-compatibility with existing scripts. New work should use
> the unified `gibson component register --kind <kind>` form above.

If registration fails, see
[Component bootstrap & auth](/docs/component-bootstrap#troubleshooting-a-401)
for the layered checks.

## 6. Verify with `inspect`

```bash
gibson inspect              # auto-detects the local credential
gibson inspect --kind agent --name my-first-agent
gibson inspect --json       # raw output, scriptable
```

`inspect` exchanges your credential for a token, calls Gibson's
identity service, and prints who you are along with the permissions you
hold. If `inspect` succeeds, your credential is healthy and you can
move on.

## Storing credentials safely

Credential files live in `~/.gibson/` and are restricted to mode `0600`
by `register`. Don't commit them. For CI and production deployments:

- **Agents and tools**: pass `client_id` and `client_secret` as
  environment variables (`OIDC_CLIENT_CREDENTIALS_CLIENT_ID`,
  `OIDC_CLIENT_CREDENTIALS_CLIENT_SECRET`,
  `OIDC_CLIENT_CREDENTIALS_TOKEN_URL`) instead of writing a file. The
  SDK picks them up automatically.
- **Plugins**: bake the host key into the deployment artifact, or
  re-register on each fresh runtime instance using a freshly-issued
  bootstrap token.
- **Workload identity** (e.g. SPIFFE): if your runtime exposes a
  workload-identity socket, the SDK auto-detects it and uses mTLS
  alongside the bearer credential. No extra configuration needed.

## Multi-environment setups

If you operate against more than one Gibson tenant (e.g. staging +
production), point `GIBSON_URL` at the right tenant per shell session
or per CI job, and keep credentials separated by passing
`--credentials-path` to `gibson` or by setting the SDK option
`agent.WithCredentialsPath(...)` from your component code.
