---
title: Secrets Management
description: Store credentials, configure your secrets backend, and bind secrets to plugins — all from the dashboard.
---

Gibson stores credentials (API keys, tokens, passwords) on your behalf so
that your plugins can retrieve them securely at runtime, without those
values ever appearing in agent logs, tool outputs, or audit streams.

This guide walks through the everyday tasks a tenant admin performs: adding
secrets, connecting a custom backend, registering a plugin, rotating
credentials, reviewing audit logs, and understanding active grants.

---

## Adding your first secret

Navigate to **Settings > Secrets**. If your account was just created,
you will see a confirmation that your secrets backend is ready.

1. Click **Add secret**.
2. Fill in the fields:
   - **Name** — a short identifier you will reference in your plugin
     manifest. Use the convention `cred:<purpose>`, for example
     `cred:openai_api_key` or `cred:db_password`.
   - **Value** — paste your credential here. The field is masked; the
     value is sent directly to secure storage and is never displayed again.
   - **Category** — choose `cred` for API keys and passwords, or
     `provider_config` for LLM provider configuration (used when wiring
     an LLM key through the secrets system).
3. Click **Save**. You are redirected to the secrets list, which shows
   the secret name, category, version number, and the last time it was
   accessed — but never the value itself.

**Can I see the value later?** No. Gibson stores credentials in encrypted
form. If you need to replace a value, use the Rotate action. There is no
"reveal value" button anywhere in the dashboard by design.

## Connecting your own secrets backend (BYO broker)

By default, new accounts use **Gibson-managed secrets storage** — a
secure storage backend provisioned for your account at signup. No
configuration is needed.

If your organization's security policy requires credentials to live in
your own infrastructure, you can switch to a bring-your-own (BYO) backend
on the **Settings > Secrets backend** page. You need the **Admin** role
to make this change.

> **Before you switch:** changing backends does not move existing secrets.
> Secrets added before the switch remain in the old backend; new secrets
> go to the new backend. If you need all secrets in one place, recreate
> them after the switch.

When you save a new backend configuration, Gibson tests connectivity
first (a "probe"). The probe writes and reads a temporary value in your
backend to confirm permissions are correct. The save is only committed
if the probe passes.

### HashiCorp Vault (self-hosted)

You need a KV v2 secrets engine and a Vault auth method that your Gibson
instance can use. The recommended auth method for production is **JWT**
(Gibson presents its service identity as a JWT); **AppRole** and static
**Token** are also supported.

Required configuration fields:

| Field | Example | Notes |
|---|---|---|
| Vault address | `https://vault.example.com` | Must be reachable from Gibson's network |
| Namespace | `gibson/tenant-acme` | Vault Enterprise only; leave blank for Community Edition |
| Path prefix | `secret/data/tenant/acme/` | Community Edition: use this instead of namespace |
| KV mount | `secret` | The KV v2 mount name |
| Auth method | `jwt` | One of: `token`, `approle`, `jwt`, `kubernetes`, `aws_iam` |
| Auth credentials | (depends on method) | For `token`: the token value; for `approle`: role ID + secret ID; for `jwt`: the JWT role name |

**Common issues:**
- If the probe returns `permission denied`, verify your Vault ACL policy
  grants `create`, `read`, `update`, `delete`, and `list` on the KV path.
- If the probe returns `KV v1 mount detected`, enable a KV v2 mount in
  your Vault and update the KV mount field.
- For Community Edition: use **Path prefix** mode, not Namespace.

### AWS Secrets Manager

Gibson assumes an IAM role to access your Secrets Manager. Your IAM role
must trust Gibson's identity (provided by your Gibson account team).

Required configuration fields:

| Field | Example | Notes |
|---|---|---|
| IAM role ARN | `arn:aws:iam::123456789012:role/gibson-secrets` | The role Gibson will assume |
| External ID | `acme-gibson-prod` | Optional; required if your trust policy enforces one |
| Region | `us-east-1` | The AWS region where your secrets live |

**IAM policy required on the assumed role:**

```json
{
  "Effect": "Allow",
  "Action": [
    "secretsmanager:CreateSecret",
    "secretsmanager:GetSecretValue",
    "secretsmanager:UpdateSecret",
    "secretsmanager:DeleteSecret",
    "secretsmanager:ListSecrets"
  ],
  "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:gibson/tenant/*"
}
```

If your Secrets Manager uses a customer-managed KMS key, also add
`kms:GenerateDataKey` and `kms:Decrypt` on that key.

**Common issues:**
- `AccessDeniedException` — check that the IAM role's trust policy allows
  Gibson's principal, and that the role's permission policy covers
  `gibson/tenant/*` secrets.
- `failed to assume role` — verify the role ARN is correct and the
  external ID (if used) matches exactly.

### GCP Secret Manager

Gibson authenticates to GCP using either a service account JSON key or
Workload Identity Federation (recommended for GKE-hosted Gibson deployments).

Required configuration fields:

| Field | Example | Notes |
|---|---|---|
| GCP project | `my-gcp-project` | The project where Secret Manager is enabled |
| Auth method | `wif` or `sa_json` | Workload Identity (recommended) or service account JSON key |
| Service account JSON | (file upload) | Only for `sa_json` auth; the key must have Secret Manager Admin rights |

Required IAM role on the service account: `roles/secretmanager.admin`
(or the narrower `roles/secretmanager.secretAdmin`).

**Before configuring:** ensure the Secret Manager API is enabled in your
GCP project:

```bash
gcloud services enable secretmanager.googleapis.com --project my-gcp-project
```

**Common issues:**
- `403 PERMISSION_DENIED` with `API not enabled` — run the command above.
- `invalid_grant` — your service account JSON key was rotated or revoked.
  Generate a new key in the GCP console and re-upload it.

### Azure Key Vault

Gibson authenticates to Azure Key Vault using a service principal (SP)
or Workload Identity.

Required configuration fields:

| Field | Example | Notes |
|---|---|---|
| Vault URL | `https://my-vault.vault.azure.net` | The full DNS name of your Key Vault |
| Azure AD tenant ID | `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` | Your Azure AD directory (tenant) ID — found in Azure Portal under Azure Active Directory > Overview |
| Client ID | `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` | The service principal's application (client) ID |
| Client secret | (masked field) | The SP's client secret; use Workload Identity to avoid managing this |
| Auth method | `service_principal` or `workload_identity` | |

The service principal needs **Get**, **Set**, and **Delete** permissions
on secrets in your Key Vault's access policies (or the **Key Vault Secrets
Officer** role if using Azure RBAC mode on the vault).

**Common issues:**
- `Forbidden 403` — check that the SP has the correct access policy or
  RBAC role on the specific Key Vault (not just at subscription level).
- `authentication failed: client secret expired` — rotate the SP secret
  in Azure AD and update the configuration here.
- `AADSTS700016` — the client ID or tenant ID is incorrect.

## Registering a plugin with secret bindings

Plugins are the only components in Gibson that can retrieve credential
values at runtime. When you register a plugin, you declare which secrets
it needs, and Gibson creates a secure binding between the plugin's identity
and those secrets.

### Starting the wizard

Go to **Settings > Plugins** and click **Add plugin**. The wizard guides
you through five steps.

### Step 1: Upload your plugin manifest

Paste your `plugin.yaml` content or drag in the file. The wizard
validates the schema immediately — any errors appear with the line number
and a description.

A minimal manifest looks like this:

```yaml
name: my-shodan-plugin
version: "1.0.0"
spec:
  runtime: process
  secrets:
    - name: cred:shodan_api_key
      rotation: live
  methods:
    - name: search_hosts
      description: Search Shodan for hosts matching a query
```

The `spec.secrets` list declares which secrets the plugin needs. You will
bind each declared secret to an actual stored value in Step 3.

### Step 2: Server-side validation

Gibson checks the manifest against the server: it confirms your account
has a working secrets backend, and that each declared secret either
already exists in your account or can be created. Any issues are shown
with specific error messages.

### Step 3: Bind secrets

For each secret declared in `spec.secrets`, choose one of:

- **Use an existing secret** — pick from a dropdown of secrets already
  in your account that match the declared name.
- **Create a new secret** — paste the value inline. This works exactly
  like the Add Secret form and the value is handled with the same
  security guarantees.

If you create secrets inline here, they appear on your Secrets page
after registration.

### Step 4: Review and confirm

A summary shows the plugin name, declared methods, and each secret
binding. When you click **Register**, Gibson atomically:

1. Creates a dedicated identity (service account) for this plugin.
2. Records the permission that allows this plugin's identity to retrieve
   each bound secret — and only those secrets.
3. Issues a one-time bootstrap token.

If anything fails, the entire registration rolls back. No partial state
is left behind.

### Step 5: Register your plugin host

You will see a bootstrap token and the CLI command to run on the machine
where your plugin will run:

```bash
gibson component register --kind plugin --token eyJ...
```

Run this command in the directory containing your `plugin.yaml`. It
exchanges the token for a persistent host key that your plugin process
will use on every subsequent connection. The token is single-use and
expires shortly after issuance (see the dashboard for the exact TTL on
your tenant).

After registration, start your plugin:

```bash
gibson component run
```

## Rotating secrets

Rotation replaces a secret's value while preserving its name, bindings,
and the plugins that use it.

### When to rotate

- A credential was exposed (breach response).
- Your security policy requires periodic rotation (for example, every 90
  days for API keys).
- A service provider forces rotation (for example, an OAuth token expired).

### How to rotate

1. Go to **Settings > Secrets**.
2. Click the secret's name to open the detail page, then click **Rotate**.
   Or click the kebab menu on the list row and choose **Rotate**.
3. Paste the new value and confirm.

The version counter increments immediately. Plugins are notified based on
the `rotation` policy declared in their manifest:

| Policy | What happens |
|---|---|
| `live` | The plugin's credential cache is invalidated. The next time the plugin requests the credential, it receives the new value — no restart needed. |
| `restart` | The plugin process exits cleanly after draining any in-flight work, then restarts and loads the new value. You may see a brief (~5 second) unavailability window while the plugin restarts. |

### What plugins are affected

The secret's detail page lists every plugin that has a binding to it
under **Plugin associations**. The Rotate confirmation modal shows the
same list so you can plan for any expected restarts before confirming.

### What if a plugin is unavailable during rotation?

For `live` rotation: the plugin will pick up the new value on its next
credential request. No action needed.

For `restart` rotation: if the plugin process is not running when
rotation occurs, it will pick up the new value when it next starts.
The restart is triggered by the plugin SDK when it reconnects to Gibson
and receives the rotation notification.

## Reviewing audit logs

Every secret operation is recorded in the audit log. Go to **Settings >
Audit log** (or the audit log link from the navigation menu).

### What is logged

| Event | When it appears |
|---|---|
| `secret.create` | A secret was added |
| `secret.read` | A plugin retrieved a credential value (the value itself is never logged — only the secret's name) |
| `secret.update` | A secret was rotated |
| `secret.delete` | A secret was deleted |
| `secret.bind` | A plugin was granted access to a secret |
| `secret.revoke_access` | A plugin's access to a secret was removed |
| `secret.config_set` | The secrets backend configuration was changed |
| `plugin.register` | A plugin was registered |
| `plugin.invoke` | A plugin method was called by a tool |
| `authz.deny` | A request was denied (for example, a plugin tried to access a secret it wasn't bound to) |

### Filtering the log

Use the **Event type** dropdown to filter by category. Combine it with
the **Date range** and **Actor** filters to narrow results.

Example: to see all credential reads in the last 7 days:
- Event type: `secret.read`
- Date range: last 7 days

Example: to find who added the `cred:openai_api_key` secret:
- Event type: `secret.create`
- Actor: (leave blank to see all actors)

Then look for the row with resource name `cred:openai_api_key`.

### Exporting for compliance

Click **Export** (CSV or JSON) to download the filtered log. Each row
includes the event type, timestamp, actor, resource name, and outcome.
**Credential values are never included in export files.**

## Understanding grants

A grant is a time-limited authorization that Gibson issues to a component
(agent, tool, or plugin) at the start of a mission. It specifies exactly
what actions that component is allowed to take and expires when the
mission ends (or after a fixed window, whichever comes first).

### Where to find them

Go to **Settings > Grants** to see all active grants in your account.

### What the columns mean

| Column | Meaning |
|---|---|
| Recipient | The specific component instance that holds the grant |
| Type | `agent`, `tool`, or `plugin`. Only plugins hold grants that cover credential retrieval. |
| Allowed actions | The list of operations this grant authorizes |
| Expires | When the grant stops being valid |

### Why grants expire

Grants are intentionally short-lived. A grant that expired cannot be
used even if it is somehow replayed. This limits the blast radius of
a compromised or misbehaving component: at most, it can act within its
granted scope until the grant expires.

If you see a grant that seems unexpectedly long-lived, check the mission
that issued it. Some missions are designed for long-running tasks and
receive correspondingly longer grants.

### Near-expiry highlighting

Rows highlighted in amber are within 5 minutes of expiry. This is
informational — the component can still act on the grant until expiry.
Gibson will issue a new grant when the next mission dispatch occurs.

### Can I revoke a grant?

The Grants page is read-only in the current release. Active missions
manage their own grant lifecycles. If you need to stop a mission
immediately, use the mission control actions on the **Missions** page,
which will cause the associated grants to expire.

## What gets logged about secrets accessed during a mission

On any mission's detail page, a **Secrets accessed** panel lists which
secret names were retrieved by plugins during that mission — the names
only, never values. This is useful for correlating mission outcomes with
specific credentials:

- Which plugin retrieved which credentials
- How many times each was retrieved
- The first and last access time within the mission

Each entry links to the full secret detail page where you can see
metadata, plugin associations, and the broader audit history.

## Deleting a secret

Before deleting a secret, check which plugins are bound to it. The
Delete confirmation dialog lists affected plugins. Deleting a secret:

1. Removes the credential value from storage immediately.
2. Removes all plugin bindings.
3. Leaves plugins that used the secret in a **Degraded** state — they
   will continue to serve methods that do not require the deleted
   credential, but methods that needed it will fail until the plugin is
   re-registered with a replacement binding.

If you need to temporarily disable a credential without removing plugin
bindings, rotate it to a known-invalid value instead of deleting it.
