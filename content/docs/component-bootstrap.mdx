A "component" in Gibson is the umbrella for **agents**, **tools**, and
**plugins**. They register with the daemon's component registry, are
addressed by name, and are dispatched by `dispatch_mode`. They share
one wire-level identity model — **all calls land on Envoy with a Bearer
JWT** — but they take **two different routes** to mint that JWT,
because the audiences and lifecycles are different.

## The two enrollment paths

| Kind | Bootstrap | Long-lived credential | Per-call token |
|---|---|---|---|
| **Plugin** | One-shot bootstrap token, single-use, 24 h expiry | Persistent **Ed25519 host key** at `~/.gibson/plugin/<name>/host_key` (mode 0600) | Plugin signs a fresh **agent+jwt** every RPC (55 s expiry) |
| **Agent / Tool** | Dashboard-issued **OAuth2 `client_id` + `client_secret`** (one-time) | Credentials JSON at `~/.gibson/<kind>/credentials` (mode 0600) | OAuth2 `client_credentials` grant returns a JWT (~1 h expiry, refreshed 5 min before TTL) |

Why two paths? Plugins are admin-installed service integrations with
manifest-declared secret bindings — the bootstrap-token-then-host-key
pattern matches that workflow, and per-RPC ephemeral signing minimizes
replay window. Agents and tools are user-built executables that need
**standard OIDC compatibility** so third-party SDKs and frameworks can
talk to the platform — OAuth2 `client_credentials` is the right protocol
for that audience.

The dashboard's **deploy wizard** at `/dashboard/deploy` handles both
paths via a per-type dispatcher. Pick the kind in step 1, the wizard
routes you to the right flow:

- `kind=plugin` delegates to the plugin-register wizard (manifest upload
  + secret bindings + atomic register, ending in a bootstrap token + a
  `gibson-cli plugin enroll --token <T>` command).
- `kind=agent` or `kind=tool` walks you through type/name selection,
  the **Permissions step** (per-action grants on components, plus
  plugin invocation grants for tools), credential issuance, and a
  polling step that confirms the component connects.

## Per-call auth handshake

Every component-to-daemon RPC takes the same shape on the wire:

```
component
  └─▶ Bearer JWT in `authorization` header
       └─▶ Envoy (jwt_authn filter, plus cgjwt verifier for plugin path)
             └─▶ ext-authz (emits `x-gibson-identity-*` headers)
                   └─▶ daemon (FGA Check + handler)
```

Two parallel JWT validators run inside Envoy:

- **Zitadel JWTs** for agents/tools — validated by Envoy's `jwt_authn`
  filter against Zitadel's JWKS (issued by `oauth2.TokenSource` from
  the SDK's `core/sdk/auth/oidc/agent_credentials.go`).
- **Capability-grant JWTs** for plugins — validated by ext-authz's
  `cgjwt.Verifier` against the daemon's published JWKS (one entry per
  enrolled host key). See `core/sdk/capabilitygrant/`.

Both paths emit the same `x-gibson-identity-*` headers downstream, so
the daemon's FGA Check doesn't care which path the caller took.

## Glossary

| Term | What it is |
|---|---|
| **Bootstrap token** | Single-use credential the plugin path mints during dashboard register. 24 h expiry. Consumed once by `gibson-cli plugin enroll`, then discarded. |
| **Host key** | Persistent Ed25519 keypair stored on disk at `~/.gibson/plugin/<name>/host_key` (0600). Identifies a plugin install. Public-key thumbprint is the host_id. |
| **Agent key** | Ephemeral Ed25519 keypair regenerated per-process. Signs the per-RPC agent+jwt; chained to the host key in the JWT's `cnf` claim. |
| **client_id / client_secret** | OAuth2 credentials the dashboard issues for agents/tools via `TenantAdminService.CreateAgentIdentity`. The secret is shown exactly once. |
| **Capability-grant JWT (CG-JWT)** | Per-mission task JWT minted by the daemon, scoped to a specific RPC set. Presented alongside the primary credential. ext-authz can short-circuit FGA when the CG-JWT covers the requested method. |
| **Component grant (FGA)** | Per-action read / configure / execute permission on a registered component, written as an FGA tuple (`component_<action>_enabled`). Editable via the dashboard's Permissions tab. |
| **Plugin invocation grant (FGA)** | Binary `can_invoke` permission on a plugin. Only **tools** can hold this; agents do not directly invoke plugins. |

## Troubleshooting a 401

1. **Verify the credential file is intact.** Mode must be 0600.
   `oidc.LoadAgentCredentials` rejects looser permissions with
   `ErrInsecureCredentialFile`.
2. **Run `gibson-cli inspect`.** Auto-detects the local credentials,
   mints a JWT, calls `IdentityService.WhoAmI`, and prints the
   principal's effective grants. If `inspect` fails at the token
   exchange, the IdP rejected the credential — re-issue from the
   dashboard.
3. **Check Envoy's `jwt_authn` logs** for the relevant route. A
   missing `aud` or invalid signature shows here before reaching
   ext-authz.
4. **Confirm ext-authz's identity headers downstream.** When the
   daemon logs `denied user:<numeric>`, it's an FGA decision —
   inspect the principal's grants in the dashboard's Permissions
   tab on the agent / tool detail page.
