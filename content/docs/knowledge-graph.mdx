Every Gibson tenant has a **knowledge graph**: a durable, queryable
record of everything your agents and tools have discovered. Hosts,
ports, services, findings, attack patterns, the missions that produced
each one, and the relationships between them.

The graph is the single biggest reason Gibson missions get smarter over
time. Today's agent doesn't repeat yesterday's recon. Today's triage
agent reads similar prior findings before deciding what to flag. None
of that requires you to write retrieval code — Gibson maintains the
graph for you, and the SDK exposes typed queries against it.

## What's in the graph

The graph stores **entities** (nodes) and **relationships** (edges).
The standard entity types Gibson ships with are described in
[Taxonomy](./taxonomy). At a high level:

- **Asset entities** — hosts, ports, services, endpoints, domains,
  subdomains, technologies, certificates.
- **Security entities** — findings, evidence, attack patterns,
  techniques.
- **Provenance entities** — missions, mission nodes, agent executions,
  decisions.
- **Custom entities** — anything outside the standard taxonomy goes
  here.

Standard relationships connect them: a host **has port**; a port
**runs service**; a finding **affects** a service; an attack pattern
**uses technique**; everything was **discovered in** a particular
mission and **produced by** a particular agent execution.

## How the graph is populated

There are three paths in:

### 1. Auto-population from tool output

Any tool that puts a `gibson.graphrag.v1.DiscoveryResult` in **field
100** of its response proto gets free graph ingestion. Gibson reads
field 100 after every tool call and creates the corresponding nodes
and relationships automatically. No graph code in your tool.

```protobuf
message Response {
  repeated Row rows = 1;
  gibson.graphrag.v1.DiscoveryResult discovery = 100;  // free ingestion
}
```

This is the recommended path for the bulk of discovery data. See
[Tools](./tools) for the full pattern.

### 2. Findings submitted by agents

When your agent calls `h.SubmitFinding(...)`, Gibson stores the finding
as a `:finding` node and links it via `AFFECTS` to the target asset.
See [Findings](./findings).

### 3. Manual writes from agents

For shapes the standard taxonomy doesn't cover, an agent can write
nodes directly:

```go
err := h.StoreNode(ctx, agent.StoreNodeRequest{
    Label: "decision",
    Properties: map[string]any{
        "reason":    "skipped this CVE — vendor disputed it",
        "cve_id":    "CVE-2024-...",
    },
    Relationships: []agent.Relationship{
        {Type: "PART_OF", To: h.Mission().Id},
    },
})
```

Use this sparingly. Auto-populating via `DiscoveryResult` keeps your
data shape consistent with the rest of the platform; ad-hoc nodes are
harder for the dashboard and other agents to reason about.

## Provenance

Every node Gibson ingests is automatically linked back to:

- the **mission** it was discovered in
- the **agent execution** that produced it
- the **target reference** it pertains to

You don't write provenance code — it's stamped on every node by the
ingestion pipeline. This is what powers the dashboard's "show me
everything this mission discovered" view, and what lets `Replay from
checkpoint` know which downstream nodes to invalidate when you replay
a step.

## Querying the graph from an agent

The SDK exposes typed graph reads via the harness. None of these
require Cypher or graph-database familiarity.

### Free-form queries

```go
hosts, err := h.QueryNodes(ctx, agent.QueryNodesRequest{
    Labels:  []string{"host"},
    Filters: map[string]any{"target_ref": h.Target().Id},
})
```

Filter by label, properties, and relationships. Useful when you want a
specific slice of the graph.

### Similarity search

The two big ones — both backed by embedding similarity over your
tenant's history:

```go
similar, err := h.FindSimilarAttacks(ctx, agent.SimilarityRequest{
    Description: "credential stuffing on web login",
    Limit:       5,
})

prior, err := h.FindSimilarFindings(ctx, agent.SimilarityRequest{
    Description: "exposed internal admin endpoint",
    Limit:       5,
})
```

Use these to ground LLM prompts ("here are five times we've seen
something like this before") or to de-duplicate before submitting a
finding.

### Attack chain walks

```go
chains, err := h.GetAttackChains(ctx, agent.ChainQuery{
    StartNodeId: someHost.Id,
    MaxDepth:    4,
})
```

Walks attack-pattern relationships from a starting node — useful for
narrative reports ("here's the path from this exposed service to the
data we care about").

## How agents use the graph in practice

Common patterns:

| Pattern | Pseudocode |
|---|---|
| **De-duplicate before submitting a finding** | `prior := h.FindSimilarFindings(...); if any(prior, similarEnough) { skip }` |
| **Ground a triage prompt with context** | `nodes := h.QueryNodes(target=current); pass to LLM as evidence` |
| **Skip recon you've done recently** | `existing := h.QueryNodes(host=target, mission_started_after=24h ago); if non-empty: re-use` |
| **Cross-mission learning** | `tactics := h.FindSimilarAttacks(scenario); use as few-shot examples` |

## Dashboard views

The dashboard surfaces graph data wherever it's contextually useful:

- **Mission detail page** — every node ingested during the run is
  listed, grouped by entity type, with click-through to the entity
  detail page.
- **Asset pages** — for any host / service / endpoint, see every
  finding ever filed against it, every mission that touched it, and
  every attack pattern that's been linked to it.
- **Findings detail** — shows the affected asset, similar prior
  findings, and the attack chain that led to it.
- **Search** — global search across the graph, scoped to your tenant.

## Per-tenant isolation

Your tenant's graph is yours. No cross-tenant reads, ever.
`FindSimilarAttacks` and `FindSimilarFindings` only consider your own
history; embeddings are not pooled across customers.

## What about retention?

Default retention is unlimited within a tenant. If you want to evict
older data — for compliance reasons, or just to keep the graph
focused — tenant admins can set retention policies under
**Settings → Knowledge graph**.

Evicted nodes are removed from the graph entirely, including their
embeddings. There is no "soft delete" path.
