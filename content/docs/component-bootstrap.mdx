title: Component bootstrap & auth
description: How agents, tools, and plugins authenticate against Gibson, the one enrollment path, the per-call protocol, and where to look when a 401 lands.
---

A "component" in Gibson is the umbrella for **agents**, **tools**, and
**plugins**. They register with Gibson by name and are addressed by name
at dispatch time. They share **one identity model** end to end: every
kind enrolls the same way, and every call to Gibson carries a Bearer
JWT minted from the same machinery. What differs between an agent, a
tool, and a plugin is **policy**, the grants on its principal, not the
mechanism.

## The enrollment path

Enrollment is a single **capability-grant** handshake for every kind:

| Step | What happens |
|---|---|
| **Mint** | `gibson agent enroll --name <name> --kind <kind>` provisions the machine identity and prints a single-use **bootstrap token** (24 h expiry, shown once). |
| **Register** | `gibson component register --token <T>` exchanges the bootstrap token for a persistent **host key** and a **runtime credential**, written to `~/.gibson/<kind>/<name>.host_key` and `~/.gibson/<kind>/<name>.runtime.json` (both mode 0600). |
| **Per-call** | The SDK signs a fresh short-lived token per RPC, chained to the host key, and presents it as a Bearer credential. Gibson validates the signature against the host key recorded at enrollment. |

The bootstrap token is consumed exactly once. Re-running `register` with
the same install is a no-op success; the host key lets the component
re-authenticate on restart without consuming another bootstrap token.

The dashboard's **deploy wizard** at `/dashboard/deploy` drives this for
all three kinds. Pick the kind in step 1:

- `kind=plugin` walks through manifest upload (declared methods and
  secret bindings) and atomic register, ending with a bootstrap token.
- `kind=agent` or `kind=tool` walks through type/name selection, the
  **Permissions** step (per-action grants on components, plus plugin
  invocation grants for tools), and a polling step that confirms the
  component connects.

Every path ends the same way: a bootstrap token and a
`gibson component register --token <T>` command to run on the host
you'll deploy from.

## Per-call protocol

Each component-to-Gibson RPC is a standard authenticated gRPC call. The
component signs a fresh short-lived token per RPC with its host key and
sends it as a Bearer credential; Gibson validates the signature against
the host key recorded at enrollment. The short per-RPC lifetime
minimises the replay window.

The SDK handles all of this transparently, your code calls
`agent.New(...)` (or the equivalent for tools and plugins), passes a
context, and Gibson sees a fully-authenticated request. You do not write
any of the token plumbing yourself.

For headless deployments (CI, your container runtime) where there is no
interactive `register` step, set `GIBSON_AGENT_KEY` (the base64-encoded
runtime credential) and `GIBSON_URL` from a secret instead of relying on
the on-disk files. The SDK reads either source.

## Glossary

| Term | What it is |
|---|---|
| **Bootstrap token** | Single-use credential the deploy wizard mints at enrollment. 24 h expiry. Consumed once by `gibson component register`, then discarded. Issued the same way for every component kind. |
| **Host key** | Persistent Ed25519 keypair stored on disk at `~/.gibson/<kind>/<name>.host_key` (mode 0600). Identifies a component install. Its public-key thumbprint is the host_id. |
| **Runtime credential** | The signing material written alongside the host key at `~/.gibson/<kind>/<name>.runtime.json` (mode 0600), or supplied as `GIBSON_AGENT_KEY` for headless runtimes. |
| **Capability-grant token (CG-JWT)** | Per-mission task token Gibson mints, scoped to a specific RPC set. Presented alongside the primary credential during mission execution. |
| **Component grant** | Per-action read / configure / execute permission on a registered component, granted via the dashboard's Permissions tab. This is the policy that makes an agent an agent and a tool a tool, the enrollment mechanism is identical. |
| **Plugin invocation grant** | Binary `can_invoke` permission on a plugin. Only **tools** can hold this; agents do not directly invoke plugins. |

## Troubleshooting a 401

A 401 means Gibson rejected your component's credential. The customer
debug surface for this is the **dashboard** plus `gibson inspect`, you
should not need to read any server-side logs.

1. **Verify the credential files are intact.** Mode must be 0600 on both
   the host key and the runtime credential. The SDK refuses to load a
   credential with looser permissions and surfaces a clear error.
2. **Run `gibson inspect`.** Auto-detects the local runtime credential,
   calls Gibson's identity service, and prints who you are along with
   the permissions you hold.
3. **If `inspect` fails to authenticate**, your enrollment never
   completed or the host key is gone, re-issue a bootstrap token from
   the dashboard's deploy wizard and re-run `gibson component register
   --token <T>`. (Bootstrap tokens are single-use and expire after 24 h;
   if you lost one or it expired, issuing a fresh one is the path
   forward.)
4. **If `inspect` succeeds but a specific RPC still gets denied**, you
   have a credential but not the grant the RPC needs. Open the relevant
   component's detail page in the dashboard and check the **Permissions**
   tab, the missing grant will be listed there with a one-click toggle
   to add it (subject to your own role's permissions to edit grants).

If you've worked through these and the call still fails, the enrollment
flow has misfired, file a bug with the output of `gibson inspect --json`
so we can correlate it on our side.
