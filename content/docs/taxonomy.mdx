Gibson ships a **standard taxonomy** of entities and relationships
(the SDK's `core.yaml`). Every mission, every tool output, every
finding references the same set of well-known types. That consistency
is what lets the dashboard, the knowledge graph, and downstream agents
reason about data they didn't produce themselves.

This page lists what Gibson recognises out of the box and explains
how to add your own node types and relationships via `taxonomy.yaml`.

For hierarchy, equivalence, and identity mappings — *"`CWE-79` is a
kind of injection"*, *"our `LeakedSecret` category is equivalent to
`gibson:disclosure`"* — see [Ontology](/docs/ontology).

## Standard entity types

Asset entities — what the target system is and exposes:

| Entity | What it represents |
|---|---|
| **Host** | A reachable host: IP address, hostname, or both. |
| **Port** | A port on a host. |
| **Service** | A service identified on a port (SSH, HTTP, …). |
| **Endpoint** | A specific URL or API endpoint exposed by a service. |
| **Domain** | A registrable domain (`example.com`). |
| **Subdomain** | A subdomain of a registered domain (`api.example.com`). |
| **Technology** | A software / framework / library identified on a target (`nginx 1.25.0`, `Spring Boot 3.x`). |
| **Certificate** | An X.509 certificate observed on a service. |

Security entities — what's interesting about the target:

| Entity | What it represents |
|---|---|
| **Finding** | A security finding. Severity, evidence, target, description. The output that matters. |
| **Evidence** | A structured payload supporting a finding (request/response, config, trace). |
| **Attack pattern** | A reusable attack pattern (e.g. "credential stuffing"). Often crosswalked to MITRE ATT&CK. |
| **Technique** | A specific technique used by an attack pattern. |

Provenance entities — who did what, when:

| Entity | What it represents |
|---|---|
| **Mission** | A mission run. |
| **Mission node** | A node within a mission run. |
| **Agent execution** | One `Execute` call by an agent during a mission. |
| **Decision** | A reasoned decision an agent or the orchestrator made (with rationale). |

Compliance entities (emitted by Gibson, not your components):

| Entity | What it represents |
|---|---|
| **Compliance signal** | A normalised marker tying observed behaviour to a compliance framework (PCI, SOC 2, HIPAA, …). |

The authoritative field-by-field schema lives in the SDK's
[`taxonomy/core.yaml`](https://github.com/zeroroot-ai/sdk/blob/main/taxonomy/core.yaml).
Don't duplicate it in your `taxonomy.yaml`.

## Standard relationships

| Relationship | From → To | Means |
|---|---|---|
| `HAS_PORT` | host → port | The port was found on the host. |
| `RUNS_SERVICE` | port → service | The service was identified on the port. |
| `AFFECTS` | finding → (host \| service \| endpoint \| …) | The finding applies to this asset. |
| `USES_TECHNIQUE` | attack pattern → technique | Composition. |
| `DISCOVERED_IN` | (any) → mission | Provenance — which mission produced this node. |
| `PRODUCES` | agent execution → (any) | An agent execution emitted this node. |
| `SIMILAR_TO` | (any) → (any) | Embedding-similarity edge written by the platform. |
| `DEPENDS_ON` | mission node → mission node | Mirrors mission DAG edges in graph form. |
| `PART_OF` | (any) → (parent) | Containment (e.g. a subdomain is `PART_OF` a domain). |
| `EMITTED_SIGNAL` | agent execution → decision (or signal) | Audit-friendly trace of why something happened. |

## Where you populate the taxonomy

The bulk of nodes come from tool output. Reserve **field 100** of
your tool's response proto for a `gibson.graphrag.v1.DiscoveryResult`
— Gibson ingests it with the right labels, relationships, and
provenance. From an agent, `h.SubmitFinding(...)` stores a finding
linked via `AFFECTS` to its target. See [Findings](/docs/findings) and
[Knowledge graph](/docs/knowledge-graph).

## Extending the taxonomy

When you need a node or relationship the standard set doesn't cover
— a `secret_finding` with team-specific fields, a `MANAGES` edge
between hosts and IoT devices — declare it in a **`taxonomy.yaml`**
at the root of your component repo.

The build flow:

1. You write `taxonomy.yaml` (custom node types and relationships).
2. `gibson component build` validates it against the SDK's
   `core.yaml` (no name collisions, properties typed correctly,
   parent/category references resolve).
3. The ADK emits `gen/taxonomy_extension.go` — a `TaxonomyExtension`
   value baked into your component binary.
4. Your component contributes the extension to the daemon at
   registration. The daemon merges it into the tenant's working
   taxonomy.
5. From then on your custom shapes are first-class in the graph,
   the dashboard, and any sibling components in the same tenant.

The same YAML shape as the SDK's `core.yaml`, scoped to your
component:

```yaml
version: "0.1.0"
kind: extension
node_types:
  - name: secret_finding
    category: security
    description: "A leaked credential discovered in source or config."
    properties:
      - name: severity
        type: string
        enum: [critical, high, medium, low]
        required: true
      - name: source_file
        type: string
        required: true
    parent: finding
    identifying_properties: [source_file, severity]

relationships:
  - name: MANAGES
    from: [host]
    to: [iot_device]
    description: "Host manages an IoT device over some control protocol."
    properties:
      - name: protocol
        type: string
```

Field reference: see the SDK's
[`taxonomy/core.yaml`](https://github.com/zeroroot-ai/sdk/blob/main/taxonomy/core.yaml)
for the full schema (property types, enums, validations, parent
chains).

## Compliance signals

If your agent identifies behaviour relevant to a compliance framework,
emit a **`ComplianceSignal`** rather than declaring a custom node.
Gibson recognises a curated list of frameworks and renders them in
the dashboard's compliance view:

```go
h.EmitComplianceSignal(ctx, agent.ComplianceSignal{
    Framework: "PCI-DSS-4.0",
    Control:   "8.3.1",
    Status:    agent.SignalFail,
    Evidence:  "TLS 1.0 detected on cardholder service.",
    TargetRef: serviceId,
})
```

To map a custom finding category to a control id, declare it in
[`ontology.yaml`](/docs/ontology) — don't reinvent the framework in
`taxonomy.yaml`.

## Runtime escape hatch

The legacy `DiscoveryResult.custom_node` and
`DiscoveryResult.explicit_relationship` fields still work for
one-shot tools that don't ship a `taxonomy.yaml`. They're useful for
quick experiments, but they bypass validation and don't surface
named types in the dashboard. Prefer `taxonomy.yaml` for anything
you intend to keep.

## Prompting your assistant

When you ask Claude / Cursor / Copilot to extend the taxonomy, give
it concrete shape. Examples:

- *"Open `taxonomy.yaml` and add a `secret_finding` node type with a
  `severity` enum (`critical|high|medium|low`) and a required
  `source_file` string. Parent is `finding`. Then run `gibson
  component build` and stop if it fails validation."*
- *"In `taxonomy.yaml`, add a `MANAGES` relationship from `host` to
  our existing `iot_device` custom node, with a `protocol` string
  property. Run `gibson component build` and surface any validator
  errors verbatim."*
- *"Read the SDK `core.yaml` and tell me whether `web_request` is
  already a standard node type before I add it to `taxonomy.yaml`."*
