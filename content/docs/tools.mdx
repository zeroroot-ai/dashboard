A **tool** is a stateless capability with a fixed input/output
contract. Agents call tools to do well-defined work — port scans, HTTP
fingerprinting, CVE lookups, AST parsers, anything where the contract
is "given X, return Y".

Tools differ from plugins in two ways:

| | Tool | Plugin |
|---|---|---|
| **State** | Stateless. Each call is independent. | Stateful. Holds connections, sessions, caches. |
| **Lifetime** | Short-lived per call. | Long-running process. |
| **Credentials** | None of its own — agents pass any data the tool needs. | Manifest-bound credentials, retrieved at runtime. |
| **Who calls it** | Agents. | Tools (and rarely, mission nodes directly). |

If your capability needs to hold a Shodan API key or maintain a
database connection, you want a [plugin](/docs/plugins). If it's "give me
the open ports on `1.2.3.4`," you want a tool.

## Tools are typed

Every tool has a **proto-defined input and output**. The contract is
explicit and discoverable — agents know exactly what they can pass in
and what they'll get back, without reading your implementation.

A tool declares:

- a fully-qualified **input message type** (e.g. `myorg.portscan.v1.Request`)
- a fully-qualified **output message type** (e.g. `myorg.portscan.v1.Response`)
- an `ExecuteProto(ctx, input) → output` function

Gibson registers your message schemas at registration time so any
agent in your tenant can dynamically marshal calls to your tool — no
shared import required.

## Auto-populating the knowledge graph

If your tool's response proto reserves field number **`100`** for a
`gibson.graphrag.v1.DiscoveryResult`, anything you put in that field is
automatically ingested into your tenant's knowledge graph during the
mission run. You don't write graph code yourself.

```protobuf
syntax = "proto3";
package myorg.portscan.v1;

import "gibson/graphrag/v1/graphrag.proto";

message Response {
  // Tool-specific fields use 1..99.
  repeated PortScanRow rows = 1;

  // Reserved for the standard discovery container — auto-ingested into
  // the knowledge graph. Optional, but highly recommended.
  gibson.graphrag.v1.DiscoveryResult discovery = 100;
}
```

`DiscoveryResult` carries the standard entity types — `host`, `port`,
`service`, `endpoint`, `domain`, `subdomain`, `technology`,
`certificate`, `finding`, `evidence` — plus `custom_node` and
`explicit_relationship` for shapes the standard taxonomy doesn't cover.
See [Taxonomy](/docs/taxonomy) for the full list.

A tool that doesn't populate field 100 just won't auto-populate the
graph. That's fine for tools whose output is purely tool-specific.

## Build a tool

### 1. Define your proto

In a fresh module:

```bash
mkdir my-portscan && cd my-portscan
go mod init example.com/my-portscan
go get github.com/zero-day-ai/sdk/tool@latest
```

Define `proto/portscan.proto`:

```protobuf
syntax = "proto3";
package myorg.portscan.v1;

option go_package = "example.com/my-portscan/api/portscan/v1;portscanv1";

import "gibson/graphrag/v1/graphrag.proto";

message Request {
  string host = 1;
  string ports = 2;   // e.g. "1-1024"
}

message Response {
  repeated Row rows = 1;
  gibson.graphrag.v1.DiscoveryResult discovery = 100;
}

message Row {
  uint32 port = 1;
  string state = 2;
  string banner = 3;
}
```

Compile with `protoc` (or your preferred buf workflow). Output goes
into `api/portscan/v1/`.

### 2. Implement the tool

```go
package main

import (
    "context"
    "log"

    "github.com/zero-day-ai/sdk/tool"
    "google.golang.org/protobuf/proto"

    portscanv1 "example.com/my-portscan/api/portscan/v1"
    graphragv1 "github.com/zero-day-ai/sdk/api/proto/gibson/graphrag/v1"
)

type portScan struct{}

func (portScan) Name() string              { return "port-scan" }
func (portScan) Version() string           { return "0.1.0" }
func (portScan) Description() string       { return "TCP connect-scan a host." }
func (portScan) Tags() []string            { return []string{"recon", "network"} }
func (portScan) InputMessageType() string  { return "myorg.portscan.v1.Request" }
func (portScan) OutputMessageType() string { return "myorg.portscan.v1.Response" }

func (p portScan) ExecuteProto(ctx context.Context, in proto.Message) (proto.Message, error) {
    req := in.(*portscanv1.Request)
    out := &portscanv1.Response{}

    // ... do the scan, populate out.Rows ...

    // Optional: populate field 100 so the knowledge graph picks up
    // hosts/ports/services automatically.
    out.Discovery = &graphragv1.DiscoveryResult{
        Hosts: []*graphragv1.Host{{Address: req.Host}},
        Ports: portsFromRows(out.Rows, req.Host),
    }
    return out, nil
}

func (portScan) Health(ctx context.Context) tool.HealthStatus {
    return tool.HealthStatus{State: tool.Healthy}
}

func main() {
    if err := tool.Serve(context.Background(), portScan{}); err != nil {
        log.Fatal(err)
    }
}
```

### 3. Register the tool

```bash
gibson component register --kind tool \
  --client-id     <from-wizard> \
  --client-secret <from-wizard> \
  --name          port-scan
```

The wizard at
[`/dashboard/deploy`](https://app.zero-day.ai/dashboard/deploy) issues
the credential. See [Install](/docs/install#4-get-a-credential-from-the-dashboard).

### 4. Run

```bash
go run .
```

The tool connects to your tenant, registers its proto schema, and
waits for invocations. Leave it running.

## Calling a tool from an agent

```go
import portscanv1 "example.com/my-portscan/api/portscan/v1"

req := &portscanv1.Request{Host: "10.0.0.1", Ports: "1-1024"}
out, err := h.CallToolProto(ctx, "port-scan", req)
if err != nil {
    return agent.NewErrorResult(err), nil
}
resp := out.(*portscanv1.Response)
```

For tools that emit incrementally (long scans, fuzzers), use
`CallToolProtoStream`. For fire-and-forget work whose results you'll
collect later, use `QueueToolWork` + `ToolResults`.

When the tool responds, Gibson:

1. Returns the typed response to your agent.
2. If field 100 was populated, ingests the `DiscoveryResult` into the
   knowledge graph and emits the appropriate `discovered.*` events to
   the mission stream.

## Authorization

Whether a given agent can call a given tool is controlled by your
tenant's **Permissions** model — see [Roles & permissions](/docs/rbac).
The dashboard's deploy wizard sets up sensible defaults; you can
tighten or loosen later.

If an agent calls a tool it doesn't have permission for, the call
returns a structured error and the mission emits a `node.finished`
with the denial reason.

## Versioning a tool

Tools are versioned via their `Version()` string. Two strategies:

- **Side-by-side**: enroll `port-scan@0.2.0` alongside `port-scan@0.1.0`,
  let agents pin to a specific version, retire the old one when no
  agent references it.
- **Rolling**: re-enroll the same name; the latest registration wins.
  Use this when you've made a backwards-compatible change.

The dashboard's tool list shows each enrolled version with its proto
schemas, so reviewers can see the contract at a glance.
