`gibson` is the local binary you use to scaffold components, register
them with your tenant, validate manifests, build and run components,
submit missions, and verify credentials.

The CLI is unified under these top-level groups:

| Group | Purpose |
|---|---|
| `init` | Initialise a Gibson workspace (pins the platform URL). |
| `login` / `logout` | Sign the CLI in and out as a human, via the device flow. |
| `component` | Scaffold, build, register, validate, and run agents / tools / plugins. |
| `agent` | Provision and manage machine identities (enroll, list, revoke). |
| `connector` | Scaffold and locally run MCP connectors. |
| `mission` | Author, validate, render, and submit missions. |
| `target` | Manage targets a mission runs against. |
| `provider` | Manage LLM provider credentials. |
| `inspect` | Print who you are and what permissions your local credential carries. |
| `docs` | Emit machine-readable docs (JSON Schemas, etc.). |

Global flags available on every command:

| Flag | Default | What it does |
|---|---|---|
| `--gibson-url` | workspace / `$GIBSON_URL` | Override the Gibson endpoint for one invocation. |
| `--quiet` | off | Suppress progress output; only print errors. |
| `--json` | off | Emit machine-readable output where supported. |

## `gibson init`

Initialise a Gibson workspace. Writes `.gibson/workspace.yaml`, pinning
the platform URL so you don't pass `--gibson-url` on every command.

```bash
gibson init --gibson-url https://api.zeroroot.ai
gibson init --global --gibson-url https://api.zeroroot.ai   # write ~/.gibson/workspace.yaml
```

| Flag | Required | Notes |
|---|---|---|
| `--gibson-url` | yes | Platform URL, e.g. `https://api.zeroroot.ai`. |
| `--global` | no | Write `~/.gibson/workspace.yaml` instead of `./.gibson/workspace.yaml`. |
| `--force` | no | Overwrite an existing workspace file. |
| `--comment` | no | Optional free-form note stored in the file. |

## `gibson login` / `gibson logout`

Authenticate the CLI as a human with the OAuth 2.0 Device Authorization
Grant. `login` prints a URL and a short code, you approve in a browser,
and the session is written to `~/.gibson/auth/credentials` (mode `0600`).
Every later `gibson` command then acts as you, and the session refreshes
silently until you run `logout`.

```bash
gibson login
gibson logout
```

| Flag | Default | Notes |
|---|---|---|
| `--issuer` | discovered | Override the identity issuer (local / air-gapped setups). |
| `--client-id` | discovered | Override the public client id. |

The CLI discovers the issuer and client id from
`GET {GIBSON_URL}/.well-known/gibson-login`; the override flags are only
needed when that endpoint is unreachable.

## `gibson component`

Group for the component lifecycle. Replaces the legacy per-kind verbs.

### `gibson component init`

Scaffold a new component directory. The `--kind` flag picks the
component shape (agent / tool / plugin); the generated files differ
per kind.

```bash
gibson component init my-agent     --kind agent
gibson component init my-portscan  --kind tool
gibson component init my-shodan    --kind plugin --dir ./my-shodan
```

| Flag | Default | Notes |
|---|---|---|
| `--kind` | yes | One of `agent`, `tool`, `plugin`. |
| `--dir` | `.` | Target directory. Created if missing. |
| `--with-secret` | none | (Plugin) Pre-populate `spec.secrets` with one entry of the given name. |
| `--force` | off | Overwrite existing files in the target directory. |

For plugins the scaffold includes `plugin.yaml`, `main.go`, a sample
proto, `Makefile`, `Dockerfile`, `.gitignore`, and `README.md`. Names
must match `^[a-z][a-z0-9-]{0,61}[a-z0-9]$` (DNS-label style).

### `gibson component build`

Generate, validate, and compile the component binary in the current
directory. Reads the component manifest, runs proto codegen, then
`go build`.

```bash
gibson component build
```

This is the canonical "build my component" command. CI pipelines that
used to invoke `make build` should call `gibson component build`
instead, the ADK keeps the toolchain pinned, so the build output is
reproducible across machines.

### `gibson component register`

Exchange a single-use bootstrap token (minted by
[`gibson agent enroll`](#gibson-agent)) for a persistent Ed25519 host key
and a runtime credential, via the capability-grant handshake. Run it from
the component directory, it reads the kind and name from `component.yaml`.

```bash
gibson component register --token <bootstrap-token>
gibson component register --token -                       # read token from stdin
gibson component register --token <T> --kind plugin       # override the auto-detected kind
```

| Flag | Required | Notes |
|---|---|---|
| `--token` | yes | Single-use bootstrap token. Pass `-` to read from stdin, or set `GIBSON_BOOTSTRAP_TOKEN`. |
| `--dir` | no | Component directory containing `component.yaml`. Default `.`. |
| `--kind` | no | Override the kind (`agent`/`tool`/`plugin`); auto-detected from `component.yaml`. |
| `--name` | no | Optional install name, allows multiple credentials per kind on one host. |
| `--gibson-url` | no | Override the platform URL; falls back to env / workspace. |

What it does:

1. Runs the capability-grant Bootstrap → Discover → Register handshake.
2. Writes two files at mode `0600`: an Ed25519 host key at
   `~/.gibson/<kind>/<name>.host_key` and a runtime credential at
   `~/.gibson/<kind>/<name>.runtime.json`.
3. Calls Gibson's identity service to confirm registration.

Re-running with the same install is a no-op success. If verification
fails, the command exits non-zero. See
[Component bootstrap & auth → Troubleshooting a 401](/docs/component-bootstrap#troubleshooting-a-401).

### `gibson component validate`

Validate the component manifest in the current directory against the
schema, without contacting Gibson.

```bash
gibson component validate
```

Useful in CI to catch manifest errors before they reach the
register step.

### `gibson component run`

Run the built component binary. Equivalent to invoking the binary
directly (which calls `agent.Serve`, `tool.Serve`, or `plugin.Serve`).

```bash
gibson component run
```

| Flag | Default | Notes |
|---|---|---|
| `--manifest` | auto-detect | Use a non-default manifest path. |

Useful in development; production deployments typically invoke the
binary directly under their process manager (systemd, Kubernetes, …).

## `gibson agent`

Provision and manage machine identities for agents, tools, and plugins.
Despite the name, the group covers every kind, the kind is a `--kind`
field, not a separate command.

### `gibson agent enroll`

Provision a new machine identity and print its one-time bootstrap token.
The token is shown once, expires in 24 hours, and is consumed by
[`gibson component register`](#gibson-component-register).

```bash
gibson agent enroll --name my-first-agent --kind agent
gibson agent enroll --name my-portscan --kind tool
```

| Flag | Required | Notes |
|---|---|---|
| `--name` | yes | Identity name (kebab-case, unique within the tenant). |
| `--kind` | no | `agent` (default), `tool`, or `plugin`. |

### `gibson agent list`

List the tenant's machine identities. Filter with `--kind`.

```bash
gibson agent list
gibson agent list --kind plugin
```

### `gibson agent revoke`

Permanently revoke a machine identity by its principal id. The next call
the component makes is rejected; re-enroll to issue a fresh identity.

```bash
gibson agent revoke <principal-id>
```

## `gibson connector`

Scaffold and locally run **MCP connectors**, a connector bridges an
existing Model Context Protocol server into Gibson as a plugin, running
the vendor MCP server unmodified. See [Connectors](/docs/connectors).

```bash
gibson connector init my-bridge   # scaffold connector.yaml
gibson connector validate         # validate the manifest
gibson connector run              # run the vendor MCP server locally, list its tools
gibson connector invoke <tool>    # invoke one discovered tool locally
```

| Command | Purpose |
|---|---|
| `init <name>` | Scaffold a `connector.yaml` manifest. |
| `validate` | Validate a `connector.yaml` manifest. |
| `run` | Run the vendor MCP server locally and list its discovered tools. |
| `invoke <tool>` | Invoke one of the vendor MCP server's tools locally. |

## `gibson mission`

Group for mission authoring, validation, rendering, and submission.

### `gibson mission submit`

Submit a mission file to the daemon. The file format is auto-detected
from the extension, `.cue`, `.yaml`, `.yml`, and `.json` are all
supported.

```bash
gibson mission submit missions/recon.yaml --target example.com
gibson mission submit missions/triage.cue --target https://app.example.com
```

| Flag | Required | Notes |
|---|---|---|
| `--target` | (depends on mission) | Target reference (URL, host, k8s cluster, …). The mission may declare its own default. |
| `--variables` | no | Path to a variables file or inline `key=value` pairs. |

What it does:

1. Loads and parses the mission file (CUE / YAML / YML / JSON).
2. Validates against the `MissionDefinition` schema.
3. Submits to the daemon's `CreateMission` RPC.
4. Calls `RunMission` and streams the lifecycle events to stdout.

See [Missions](/docs/missions) for the full lifecycle.

## `gibson target`

Create, list, inspect, update, and delete the assessment targets a
mission runs against. Targets are referenced by the minted UUID.

```bash
gibson target create   # register a target, prints its UUID
gibson target list      # list targets (UUID + metadata)
gibson target get <uuid>
gibson target update <uuid>
gibson target delete <uuid>
```

---

## `gibson provider`

Manage the LLM provider configurations your agents draw on. Credentials
are stored by Gibson, never written to disk by the CLI.

```bash
gibson provider add     # create a provider configuration
gibson provider list
gibson provider test    # validate a config with a live probe
gibson provider delete <name>
```

See [Secrets management](/docs/secrets-management) for how provider
credentials are stored.

## `gibson inspect`

Print your effective identity and permissions.

```bash
gibson inspect                            # auto-detect
gibson inspect --kind agent               # specify kind
gibson inspect --kind plugin --name foo   # specify name when several plugins are registered locally
gibson inspect --json                     # machine-readable
```

| Flag | Default | Notes |
|---|---|---|
| `--kind` | auto-detect | `agent`, `tool`, or `plugin`. Required when more than one kind has a local credential. |
| `--name` | auto | Required when more than one credential exists for a single kind. |
| `--json` | off | Emit the raw identity payload. |

What it does:

1. Loads the credential.
2. Mints a JWT (or signs a capability-grant for plugins).
3. Calls Gibson's identity service to fetch your principal.
4. Prints your identity, tenant, and effective grants.

If `inspect` fails:

- **At credential load**, credential file is missing or has wrong
  permissions. Fix the file mode (`chmod 600`) or re-register.
- **At token mint**, credential is malformed. Re-register.
- **At identity call**, Gibson rejected the credential (revoked,
  expired, wrong tenant). Re-issue from the dashboard.

## `gibson docs`

Emit machine-readable docs, JSON Schemas for the component manifest,
mission file format, taxonomy types, and similar. Useful for IDE
plugins, codegen, and pinning your CI against a specific CLI version's
contracts.

```bash
gibson docs
```

## Exit codes

| Code | Meaning |
|---|---|
| `0` | Success. |
| `1` | Generic error. |
| `2` | Bad arguments / usage. |
| `3` | Credential file missing or unreadable. |
| `4` | Credential rejected by Gibson. |
| `5` | Network error reaching Gibson. |
| `75` | Plugin runtime exited cleanly to be restarted (rotation-restart sentinel). |

CI scripts can rely on these. `2` and `3` indicate operator error;
`4` and `5` indicate platform-side issues to surface.

## Environment variables

| Variable | Used by | Effect |
|---|---|---|
| `GIBSON_URL` | every command, every component | Endpoint to dial. |
| `GIBSON_AGENT_KEY` | every component kind | Inline base64-encoded runtime credential, no on-disk file needed. Pair with `GIBSON_URL`. |
| `GIBSON_BOOTSTRAP_TOKEN` | `component register` | Bootstrap token, as an alternative to the `--token` flag. |
| `GIBSON_HOME` | every command | Relocate the config dir (default `~/.gibson`). |
| `GIBSON_LOG_LEVEL` | every command | `debug` / `info` / `warn` / `error`. Default `info`. |

The env-variable path is the recommended way to ship CI workloads -
no credential files on disk.
